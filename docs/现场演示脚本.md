# 现场演示脚本

## 1. 演示目标

在 5 到 8 分钟内向评审展示系统的完整闭环：

- 上传 PDF 简历
- 自动解析简历内容
- 提取候选人关键信息
- 输入岗位 JD
- 输出匹配度评分和关键词命中结果
- 展示 AI 简短总结、阿里百炼 Key 调试入口和缓存设计

## 4. 演示流程

### 第 1 分钟：介绍项目

可以这样说：

```text
这个项目是一个 AI 赋能的智能简历分析系统，主要解决招聘场景中大量 PDF 简历筛选耗时的问题。系统支持上传 PDF 简历，自动提取候选人关键信息，并结合岗位 JD 计算匹配度评分。
```

### 第 2 分钟：展示前端页面和 Key 调试入口

操作：

- 打开线上演示地址
- 展示顶部“百炼 Key”输入框和“生成 env”按钮
- 展示上传区域、JD 输入区域和结果区域

可以这样说：

```text
前端页面主要保留三个核心操作：上传简历、输入岗位需求、查看分析结果。为了方便调试，我把阿里百炼 Key 配置也放到了页面顶部，输入后后端会自动生成 .env 文件，本地演示不需要手动改配置。
```

### 第 3 分钟：上传简历并解析

操作：

- 上传一份 PDF 简历
- 点击解析或分析按钮
- 展示姓名、电话、邮箱、地址、技能、学历、AI 简短总结

可以这样说：

```text
后端会先解析 PDF 的多页文本，再进行文本清洗。电话和邮箱会优先使用规则提取，学历、求职意向、技能和简短总结会结合阿里百炼大模型补全。学历这类复杂结构会统一转成中文可读文本，避免页面直接显示 JSON。
```

### 第 4 分钟：输入岗位 JD 并评分

操作：

- 粘贴后端开发岗位 JD
- 点击匹配分析
- 展示总分、技能分、经验分、学历分、AI 分、命中关键词、缺失关键词

可以这样说：

```text
匹配评分分为两部分。第一部分是规则评分，比如技能关键词、工作年限和学历要求；第二部分是 AI 语义评分，用来判断项目经历和岗位职责的相关性。最终分数会按权重合成，并返回可解释的评分依据。
```

### 第 5 分钟：展示缓存效果

操作：

- 对同一份简历和同一份 JD 再次点击分析
- 展示 `缓存命中：是`

可以这样说：

```text
系统会基于简历 hash 和 JD hash 做缓存。同一份简历或同一组匹配请求不会重复调用模型，可以降低成本并提升响应速度。
```

### 第 6 分钟：展示测试文件、GitHub 和 README

操作：

- 展示 `docs/test-pdfs` 中的 PDF 测试文件
- 打开 GitHub 仓库
- 展示 README、架构文档、API 文档、部署文档

可以这样说：

```text
仓库中包含项目架构、技术选型、接口说明、部署步骤、现场演示脚本和测试 PDF。当前版本已完成本地演示闭环，后续可以把后端部署到阿里云函数计算，前端部署到 GitHub Pages。
```

## 5. 常见问题回答

### 为什么选择 Serverless？

```text
这个项目的请求量不稳定，主要在上传和分析时消耗计算资源。Serverless 可以按量计费，部署成本低，也符合题目对阿里云函数计算的要求。
```

### 为什么现在用内存缓存？

```text
MVP 阶段核心目标是完成单次简历分析闭环，不涉及用户体系和长期历史记录，因此优先使用内存缓存减少重复计算。如果后续扩展多用户和历史记录，可以加入 MySQL、PostgreSQL 或阿里云 Tair。
```

### AI 提取 JSON 不稳定怎么办？

```text
我会在 prompt 中明确要求模型只返回 JSON，并在后端使用 Pydantic 做结构校验。如果模型返回异常，会进行一次修复重试；仍失败时返回基础解析结果和错误提示。
```

### 扫描版 PDF 能不能识别？

```text
普通模式和极速模式主要处理文本型 PDF。精准模式预留了视觉 OCR 模型配置，可以结合阿里百炼视觉能力进一步增强扫描版 PDF 识别。
```

## 6. 兜底方案

如果线上前端临时异常：

- 使用 Postman 或 Apifox 直接调用后端接口
- 展示 GitHub README 和接口返回 JSON

如果后端云函数临时异常：

- 本地启动后端服务
- 本地启动前端服务
- 使用本地地址完成演示

如果 AI 服务临时异常：

- 展示已缓存的分析结果
- 说明后端已做异常处理，AI 不可用时仍可返回基础 PDF 解析结果
