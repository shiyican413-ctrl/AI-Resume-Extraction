# 现场演示脚本

## 1. 演示目标

在 5 到 8 分钟内向评审展示系统的完整闭环：

- 上传 PDF 简历
- 自动解析简历内容
- 提取候选人关键信息
- 输入岗位 JD
- 输出匹配度评分和关键词命中结果
- 说明系统架构、AI 调用和缓存设计

## 2. 演示前准备

准备材料：

- 线上前端地址
- GitHub 仓库地址
- 1 份后端开发岗位 JD
- 2 份 PDF 简历，一份匹配度较高，一份匹配度较低
- Postman 或 Apifox 接口请求作为兜底

检查事项：

- 前端页面可以访问
- 后端健康检查接口正常
- 模型服务密钥未过期
- PDF 文件大小符合限制
- 浏览器控制台没有跨域错误

## 3. 演示流程

### 第 1 分钟：介绍项目

可以这样说：

```text
这个项目是一个 AI 赋能的智能简历分析系统，主要解决招聘场景中大量 PDF 简历筛选耗时的问题。系统支持上传 PDF 简历，自动提取候选人关键信息，并结合岗位 JD 计算匹配度评分。
```

### 第 2 分钟：展示前端页面

操作：

- 打开线上演示地址
- 展示上传区域、JD 输入区域和结果区域

可以这样说：

```text
前端页面主要保留三个核心操作：上传简历、输入岗位需求、查看分析结果。这样评审可以直接验证核心功能，不需要登录或复杂配置。
```

### 第 3 分钟：上传简历并解析

操作：

- 上传一份 PDF 简历
- 点击解析或分析按钮
- 展示姓名、电话、邮箱、地址、技能、学历、项目经历

可以这样说：

```text
后端会先解析 PDF 的多页文本，再进行文本清洗。电话和邮箱会优先使用正则提取，其他字段通过大模型按固定 JSON 格式抽取，保证前端可以稳定渲染。
```

### 第 4 分钟：输入岗位 JD 并评分

操作：

- 粘贴后端开发岗位 JD
- 点击匹配分析
- 展示总分、技能分、经验分、AI 分、命中关键词、缺失关键词

可以这样说：

```text
匹配评分分为两部分。第一部分是规则评分，比如技能关键词、工作年限和学历要求；第二部分是 AI 语义评分，用来判断项目经历和岗位职责的相关性。最终分数会按权重合成，并返回可解释的评分依据。
```

### 第 5 分钟：展示缓存效果

操作：

- 对同一份简历和同一份 JD 再次点击分析
- 展示响应速度变快或 `cache_hit=true`

可以这样说：

```text
系统会基于简历 hash 和 JD hash 做缓存。同一份简历或同一组匹配请求不会重复调用模型，可以降低成本并提升响应速度。
```

### 第 6 分钟：展示 GitHub 和 README

操作：

- 打开 GitHub 仓库
- 展示 README、架构文档、API 文档、部署文档

可以这样说：

```text
仓库中包含项目架构、技术选型、接口说明、部署步骤和现场演示脚本。后端按阿里云 Serverless 设计，前端部署在 GitHub Pages，便于评审线上验收。
```

## 4. 常见问题回答

### 为什么选择 Serverless？

```text
这个项目的请求量不稳定，主要在上传和分析时消耗计算资源。Serverless 可以按量计费，部署成本低，也符合题目对阿里云函数计算的要求。
```

### 为什么不用传统数据库？

```text
MVP 阶段核心目标是完成单次简历分析闭环，不涉及用户体系和长期历史记录，因此优先使用 Redis 缓存减少重复计算。如果后续扩展多用户和历史记录，可以加入 MySQL 或 PostgreSQL。
```

### AI 提取 JSON 不稳定怎么办？

```text
我会在 prompt 中明确要求模型只返回 JSON，并在后端使用 Pydantic 做结构校验。如果模型返回异常，会进行一次修复重试；仍失败时返回基础解析结果和错误提示。
```

### 扫描版 PDF 能不能识别？

```text
当前 MVP 主要处理文本型 PDF。扫描版 PDF 可以作为后续优化方向，引入 OCR，例如阿里云 OCR 或 PaddleOCR。
```

## 5. 兜底方案

如果线上前端临时异常：

- 使用 Postman 或 Apifox 直接调用后端接口
- 展示 GitHub README 和接口返回 JSON

如果后端云函数临时异常：

- 本地启动后端服务
- 本地启动前端服务
- 使用本地地址完成演示

如果 AI 服务临时异常：

- 展示已缓存的分析结果
- 说明后端已做异常处理，AI 不可用时仍可返回基础 PDF 解析结果
